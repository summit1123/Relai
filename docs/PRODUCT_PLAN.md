# Relai 제품 기획서

## 한 줄 정의

Relai는 요구사항 변경이 들어왔을 때 `어디까지 영향이 퍼지는지`와 `무엇이 확정이고 무엇이 다시 열렸는지`를 같은 화면에서 보여주는 요구사항 변경 계약 시스템입니다.

## 왜 지금 필요한가

실무 팀의 낭비는 초안 작성보다 `중간 변경`에서 커집니다. 정책, 일정, 운영 조건, 고객 요청이 뒤늦게 바뀌면 팀은 기능을 만드는 시간보다 `무엇을 다시 검토해야 하는지`를 맞추는 데 더 많은 시간을 씁니다.

Relai는 이 재작업 구간을 겨냥합니다.

- 변경이 들어오면 누가 다시 확인해야 하는지가 사람 기억에 의존합니다.
- 잠긴 계약과 열린 질문이 섞여 있어 착수 가능 여부가 흐려집니다.
- 승인권자는 이미 작업이 진행된 뒤에야 결정을 요청받습니다.

## 제품 가설

요구사항과 변경 요청을 계약 단위로 쪼개고, 각 항목의 상태를 `잠김`, `변경으로 다시 열림`, `승인 대기`, `열린 질문`, `착수 가능`으로 명확히 나누면 팀은 같은 기준으로 더 빨리 결정할 수 있습니다.

Relai의 wow point는 아래 두 요소가 첫 화면에서 동시에 읽힐 때 가장 강합니다.

- 변경 영향 맵
- 합의 잠금 보드

## 핵심 운영 화면

| 화면 | 역할 | 반드시 보여야 하는 정보 | 주 액션 |
| --- | --- | --- | --- |
| Change Control Tower | 메인 운영 화면 | 영향 맵, 합의 잠금 보드, 승인 조건, 근거 로그 | 항목 선택, 승인 브리핑 이동, 변경 요청 추가 |
| Contract Detail View | 계약 검토 화면 | 원문 근거, 역할별 재구성, 연결 영향, 책임자 | 질문 추가, 담당자 지정, 재검토 |
| Approval Briefing View | 승인 판단 화면 | 결정 대상, 보류 리스크, 차단 작업, 다음 액션 | 승인, 보류, 추가 확인 요청 |
| Change Request Intake View | 변경 접수 화면 | 요청 제목, 이유, 긴급도, 연결 계약, 요청자 | 저장, 재분석 |

## 첫 화면 정보구조

첫 화면은 아래 순서를 유지해야 합니다.

1. 상단 운영 헤더: 현재 릴리즈, 다시 열린 계약 수, 승인 필요 수
2. 좌측 영향 맵: 화면/API/데이터/테스트/일정/담당자 영향 노드
3. 중앙 합의 잠금 보드: `잠김`, `변경으로 다시 열림`, `승인 대기`, `열린 질문`, `착수 가능`
4. 우측 브리핑 레일: 지금 결정할 항목, 차단 요소, 다음 액션
5. 하단 근거 로그: 원문, 회의, 채팅, 메모리 링크

이 구성을 지키는 이유는 사용자가 첫 10초 안에 `무엇이 흔들렸는지`와 `누가 결정해야 하는지`를 읽게 하기 위해서입니다.

## 상태 전이 계약

| 상태 | 의미 | 다음으로 갈 수 있는 상태 | 운영 의미 |
| --- | --- | --- | --- |
| 빈 상태 | 아직 분석할 원문이 없음 | 분석 중 | 입력과 샘플 시작점을 먼저 제공합니다. |
| 분석 중 | 구조화와 영향 분석 실행 중 | 분석 성공, 분석 실패 | 기존 결과가 있으면 유지한 채 갱신 중으로 보여줍니다. |
| 분석 성공 | 최신 영향 맵과 잠금 보드가 생성됨 | 승인 대기, 변경으로 다시 열림 | 운영자가 현재 판단 대상을 볼 수 있습니다. |
| 분석 실패 | 호출 또는 파싱이 실패함 | 분석 중 | 실패 이유와 재시도 경로를 같이 보여줍니다. |
| 승인 대기 | 승인권자 결정이 남음 | 잠김, 보류, 추가 확인 필요 | 착수 불가 조건을 분명히 드러냅니다. |
| 추가 확인 필요 | 근거 부족 또는 영향 미확정 | 분석 중, 승인 대기 | 확정처럼 보이지 않고 확인 요청을 남깁니다. |
| 잠김 | 계약이 확정됨 | 변경으로 다시 열림 | 팀이 안심하고 착수할 수 있습니다. |
| 변경으로 다시 열림 | 새 요청으로 계약을 다시 봐야 함 | 승인 대기, 잠김, 보류 | 재작업 범위와 재합의 포인트가 드러납니다. |
| 보류 | 결정이 미뤄짐 | 승인 대기, 변경으로 다시 열림 | 일정 리스크와 차단 상태를 계속 표시합니다. |

## 사용자 시나리오

### 시나리오 A. 기획자가 초안을 등록합니다

1. 원문과 담당자를 입력합니다.
2. Relai가 계약 후보와 열린 질문을 나눕니다.
3. 첫 화면에서 영향 맵과 합의 잠금 보드를 함께 확인합니다.

### 시나리오 B. 개발 리드가 기술 관점을 되돌려 줍니다

1. Contract Detail View에서 구현 제약과 테스트 기준을 확인합니다.
2. Relai는 이를 다시 기획 언어로 재구성합니다.
3. 승인 필요 항목이 별도 열로 이동합니다.

### 시나리오 C. 변경 요청이 들어옵니다

1. Change Request Intake View에서 제목, 이유, 긴급도를 등록합니다.
2. 재분석 후 영향 맵의 흔들리는 노드가 강조됩니다.
3. 기존 잠긴 계약 일부가 `변경으로 다시 열림` 상태로 이동합니다.
4. 승인권자는 Approval Briefing View에서 승인 또는 보류를 결정합니다.

## MVP 범위

이번 MVP에 반드시 포함하는 기능은 아래와 같습니다.

1. 초안 입력과 파일 불러오기
2. 담당자 연결
3. 기획 관점 ↔ 개발 관점 재구성
4. 변경 영향 맵
5. 합의 잠금 보드
6. 승인 브리핑
7. 변경 요청 등록과 재분석
8. AI 키가 없어도 흐름이 유지되는 데모 모드

## MVP에서 뒤로 미루는 것

- Jira, Slack, Notion 실연동
- 멀티 프로젝트 운영
- 조직 권한 체계
- 정교한 버전 비교 히스토리
- 실시간 회의 보조 고도화

## UI 가드레일

- 채팅형 AI 비서 레이아웃을 메인 경험으로 사용하지 않습니다.
- 카드 나열보다 보드, 표, 상태 구획을 우선합니다.
- 상태는 색만이 아니라 열 위치, 책임자, 근거 링크로 함께 구분합니다.
- 첫 화면의 첫 스크롤 안에서 `무엇이 잠겼고 무엇이 다시 열렸는지`가 읽혀야 합니다.

## 성공 판단 기준

- 첫 화면에서 wow point가 바로 읽힙니다.
- 성공, 실패, 빈 상태, 승인 대기, 추가 확인 필요가 문서와 구현 모두에서 분리됩니다.
- 변경 요청이 단순 로그가 아니라 계약 상태를 다시 여는 이벤트로 정의됩니다.
- generic AI copilot처럼 보이지 않는 운영 정보구조가 유지됩니다.

## 이번 런의 산출물

- 읽히는 PRD와 제품 요약
- 화면 정보구조와 상태 전이 계약 문서
- 이후 UI 구현으로 바로 연결 가능한 source of truth
