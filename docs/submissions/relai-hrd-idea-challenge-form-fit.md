# 「2026 미래혁신 Idea 챌린지」 양식 맞춤 작성안

## 참가신청서 - 아이디어 핵심 요약

RelAi는 기획·개발·품질 과정에서 발생하는 요구사항 변경과 팀 논의 기록을 AI가 하나의 프로젝트 메모리로 통합하고, 이를 결정·질문·승인·영향·액션 단위로 구조화하는 AI 업무혁신 서비스입니다. 기획자는 AI로 요구사항과 사용자 시나리오를 빠르게 구체화하고, 개발자는 AI로 구현 방식과 코드 구조를 빠르게 설계하며, 품질 담당자는 AI로 테스트 케이스와 예외 상황을 확장합니다. 그러나 각자가 AI를 사용하는 목적과 방식이 달라지면서 같은 요구사항도 서로 다른 방향으로 해석될 수 있습니다. RelAi는 이러한 AI 활용 방식의 분산을 팀 단위의 합의 구조로 다시 정렬합니다.

사용자가 요구사항 초안, 변경 요청, 채팅, 회의 기록을 등록하면 RelAi는 이전 프로젝트 맥락을 함께 참고하여 무엇이 확정된 결정인지, 무엇이 아직 열린 질문인지, 어떤 항목이 승인되어야 하는지, 변경이 화면·API·데이터·테스트·일정·담당자에게 어떤 영향을 주는지 분석합니다. 또한 채팅만으로 합의가 어려운 경우 AI가 회의 필요성을 감지하고 회의 안건과 참석자를 제안하며, 회의가 끝난 뒤에는 회의 기록을 다시 프로젝트 메모리로 저장합니다. 이를 통해 요구사항 변경 검토 시간, 재작업, 승인 지연, 회의 후 결정 누락을 줄이고 AI 시대에 맞는 협업 운영 구조를 제공합니다.

## 1. 아이디어 개발 배경 및 필요성

현업에서 기획, 개발, 품질 검수, 승인 과정은 한 번에 완성된 요구사항을 순서대로 처리하는 방식으로만 진행되지 않습니다. 기획 초안이 나오면 개발이 시작되고, 개발 중간에 정책이나 우선순위가 바뀌며, 품질 검수 과정에서 다시 예외 케이스와 수정 사항이 발견됩니다. 이러한 과정은 특정 직군의 역량 부족으로만 설명하기 어렵습니다. 실제 업무에서는 요구사항이 계속 구체화되고, 이해관계자가 늘어나며, 출시 일정과 품질 기준이 함께 움직이기 때문입니다.

저는 AI 소프트웨어 개발자로 일하면서 이러한 문제를 반복적으로 경험했습니다. 기획이 나오면 개발을 진행하고, 개발 중간에 기획이 수정되며, 품질팀의 검수 과정에서 다시 개발 범위가 바뀌는 일이 발생합니다. 이때 가장 어려운 점은 변경이 있었다는 사실 자체가 아니라, 무엇이 이미 확정되었고 무엇이 다시 논의되어야 하며 어떤 개발 영역이 영향을 받는지 빠르게 파악하기 어렵다는 점입니다. 요구사항 변경의 이유와 합의 맥락은 메신저, 회의, 문서, 개인 기억 속에 흩어져 있고, 변경이 발생할 때마다 다시 확인하고 조율하는 비용이 반복됩니다.

최근 생성형 AI의 확산은 이 문제를 더 크게 만들 수 있습니다. AI는 개인의 작업 속도를 빠르게 높이고 있습니다. 기획자는 AI로 사용자 시나리오와 정책 문구를 빠르게 작성하고, 개발자는 AI로 코드와 구현 방식을 빠르게 설계하며, 품질 담당자는 AI로 테스트 케이스와 예외 상황을 더 촘촘히 도출합니다. 그러나 각자가 AI를 사용하는 목적과 방식이 다르기 때문에, 같은 요구사항도 기획 의도, 구현 범위, 품질 기준이 서로 다른 방향으로 빠르게 확장될 수 있습니다.

즉, AI 도입의 문제는 더 이상 AI를 쓰느냐 쓰지 않느냐에만 있지 않습니다. 개인 단위의 AI 활용은 늘어나고 있지만, 그 결과물을 팀 단위의 합의 구조로 다시 정렬하는 방식은 여전히 기존의 문서, 메신저, 회의 중심에 머물러 있습니다. Microsoft Work Trend Index 2024에 따르면 지식근로자의 75%가 업무에서 AI를 사용하고, AI 사용자 중 78%는 개인적으로 선택한 AI 도구를 업무에 활용하고 있습니다. Gallup 조사에서도 미국 직장인의 업무상 AI 사용은 2023년 21%에서 2025년 40%로 증가했습니다. 반면 조직 차원의 명확한 AI 전략을 전달받았다는 응답은 22%에 그쳤습니다.

개발 현장에서도 유사한 병목이 나타납니다. Atlassian의 2025 Developer Experience 조사에서는 개발자 68%가 AI로 주 10시간 이상을 절약한다고 답했지만, 동시에 개발자 50%는 정보 탐색, 도구 전환, 협업 문제 등 조직 비효율로 주 10시간 이상을 잃고 있었습니다. 또한 개발자가 실제 코딩에 쓰는 시간은 약 16%에 불과하며, 나머지 시간은 요구사항 확인, 회의, 승인, 문서 확인, 협업 조율 등 코드 주변 업무에 쓰입니다.

의사소통 문제 역시 실제 재작업으로 이어집니다. Atlassian/Loom의 2026년 조사에서는 근로자 80%가 글로 된 메시지를 오해해 불필요하거나 다시 해야 하는 일을 경험했다고 응답했습니다. PMI 2021 조사에서도 프로젝트 실패 원인으로 요구사항 수집 부실 40%, 프로젝트 목표 변경 38%, 커뮤니케이션 부실 28%가 제시되었습니다.

따라서 AI 시대의 업무혁신은 단순히 개인에게 더 많은 AI 도구를 제공하는 데서 끝나서는 안 됩니다. 기획, 개발, 품질, 승인자가 각자 AI로 만든 판단을 하나의 프로젝트 맥락 안에서 정렬하고, 변경 사항이 어디까지 영향을 미치는지 빠르게 파악할 수 있는 협업 운영 구조가 필요합니다. RelAi는 이러한 문제를 해결하기 위해 제안하는 AI 업무혁신 시스템입니다.

## 2. 아이디어의 혁신성 및 차별성

RelAi는 요구사항 문서, 팀 채팅, 회의 기록, AI 산출물을 하나의 프로젝트 메모리로 통합하고, 이를 결정·질문·승인·영향·액션 단위로 구조화하는 AI 협업 운영 시스템입니다. 기존 협업 도구가 문서, 티켓, 채팅, 회의록을 각각 저장하는 데 초점을 둔다면, RelAi는 변경이 발생했을 때 어떤 결정이 다시 열렸는지, 어떤 화면과 API가 영향을 받는지, 누가 승인해야 하는지, 이전 회의에서 어떤 조건이 합의되었는지를 하나의 흐름으로 연결합니다.

RelAi의 첫 번째 차별점은 AI 활용 방식의 분산을 팀 단위의 합의 구조로 정렬한다는 점입니다. 기획자가 AI로 만든 사용자 시나리오, 개발자가 AI로 만든 구현 판단, 품질 담당자가 AI로 만든 테스트 관점은 각각 유용하지만, 서로 연결되지 않으면 오히려 해석 차이를 키울 수 있습니다. RelAi는 이러한 산출물을 하나의 프로젝트 맥락 안에서 다시 해석하고, 같은 요구사항을 기획 관점, 개발 관점, 품질 관점으로 나누어 보여줍니다.

두 번째 차별점은 변경 영향 맵입니다. 요구사항 변경은 단순한 문장 수정으로 끝나지 않습니다. 하나의 정책 변경이 화면, API, 데이터베이스, 테스트 케이스, 일정, 담당자, 승인권자에게 연쇄적으로 영향을 줄 수 있습니다. RelAi는 변경 요청을 입력받으면 해당 변경이 어떤 업무 영역에 영향을 주는지 시각적으로 구조화합니다. 이를 통해 사용자는 변경 검토 초기에 재작업 범위와 필요한 의사결정을 파악할 수 있습니다.

세 번째 차별점은 합의 잠금 보드입니다. RelAi는 논의 내용을 단순히 저장하지 않고, 이미 합의된 결정, 아직 답이 필요한 열린 질문, 승인권자의 판단이 필요한 항목, 개발 착수 전에 필요한 조건을 분리합니다. 이를 통해 팀은 “무엇이 확정되었는지”와 “무엇이 다시 열렸는지”를 명확히 구분할 수 있습니다.

네 번째 차별점은 프로젝트 메모리입니다. 요구사항 변경은 현재 입력된 문장 하나만으로 판단하기 어렵습니다. 과거 회의에서 “이 기능은 2차 릴리즈로 미룬다”고 합의했을 수도 있고, 개발 리드가 “이 API 변경은 QA 기간 2일이 추가로 필요하다”고 말했을 수도 있습니다. RelAi는 이러한 채팅과 회의 기록을 메모리 카드로 저장하고, 새로운 변경 요청이 들어왔을 때 이전 맥락을 함께 참고합니다.

다섯 번째 차별점은 AI 회의 오케스트레이션입니다. 채팅만으로 합의가 어렵거나 기획, 개발, 품질 관점이 충돌하는 경우 RelAi는 회의 필요성을 감지하고 회의 제목, 참석자, 핵심 안건, 결정해야 할 항목을 제안합니다. 회의가 끝나면 회의 기록을 다시 분석하여 새 결정, 열린 질문, 승인 필요 항목, 액션 아이템으로 반영합니다.

이러한 구조를 통해 RelAi는 단순한 AI 채팅앱, 회의록 요약 도구, 프로젝트 관리 도구와 구분됩니다. RelAi의 핵심은 더 많은 문서를 만드는 것이 아니라, AI 시대에 더 빠르게 발생하는 요구사항 변경과 해석 차이를 팀 단위의 실행 가능한 합의 구조로 바꾸는 것입니다.

## 3. 기술 구현 방안 및 실현 가능성

RelAi는 MVP 단계에서 문서·채팅·회의 기록을 입력받아 AI가 구조화하고, 그 결과를 변경 영향 맵, 합의 잠금 보드, 프로젝트 메모리, 회의 안건으로 제공하는 웹 기반 서비스로 구현할 수 있습니다.

우선 사용자는 요구사항 초안, 변경 요청, 품질 피드백, 회의 기록을 프로젝트 단위로 등록합니다. 입력 데이터에는 작성자, 역할, 담당자, 목표 일정, 관련 링크, 회의 기록 등 기본 메타데이터가 함께 저장됩니다. 이후 AI 분석 엔진은 입력된 내용을 결정, 질문, 승인, 영향, 액션으로 분류합니다.

분류된 결과는 역할별 관점으로 재구성됩니다. 기획 관점에서는 사용자 가치, 정책 의도, 운영 조건을 중심으로 정리하고, 개발 관점에서는 구현 범위, API 영향, 예외 처리, 테스트 조건을 중심으로 정리합니다. 품질 관점에서는 검증해야 할 시나리오, 예외 케이스, 회귀 테스트 가능성을 중심으로 정리합니다. 이를 통해 같은 요구사항을 각 역할이 이해할 수 있는 실행 언어로 전환할 수 있습니다.

다음으로 변경 영향 맵을 생성합니다. 변경 요청이 입력되면 AI는 해당 변경이 화면, API, 데이터, 테스트, 일정, 담당자, 승인권자에게 미치는 영향을 추론합니다. MVP 단계에서는 실제 코드 저장소 분석까지 자동화하지 않고, 요구사항 문서와 사용자가 제공한 시스템 정보를 바탕으로 영향 범위를 제시합니다. 향후에는 GitHub, GitLab, Jira, Notion, Slack, Teams와 연동하여 실제 업무 데이터와 코드 변경 이력까지 분석 범위를 확장할 수 있습니다.

합의 잠금 보드는 변경 요청과 논의 결과를 상태별로 관리합니다. 항목은 잠긴 결정, 열린 질문, 승인 필요, 착수 가능 조건으로 구분됩니다. 이 보드는 프로젝트 관리 도구의 일반적인 할 일 목록과 다르게, “무엇을 해야 하는가”보다 “무엇이 합의되었고 무엇이 아직 합의되지 않았는가”를 중심으로 보여줍니다.

프로젝트 메모리는 RelAi의 핵심 기술 구조입니다. 회의 기록, 채팅, 의사결정, 기술 제약, 반복되는 품질 이슈는 메모리 카드로 저장됩니다. 새로운 변경 요청이 들어오면 AI는 현재 입력뿐 아니라 이전 메모리를 함께 참고하여 과거 합의와 충돌하는지, 다시 검토해야 할 항목이 있는지 판단합니다.

회의 오케스트레이션 기능은 채팅과 보드의 상태를 기반으로 동작합니다. AI가 충돌 신호, 미결정 승인 항목, 반복되는 질문, 일정 영향이 큰 변경을 감지하면 회의 개설을 제안합니다. 이때 회의 제목, 참석 대상, 핵심 안건, 사전 확인 자료, 결정해야 할 항목을 자동으로 정리합니다. 회의 후에는 회의록 또는 녹음 기록을 다시 분석하여 보드와 메모리에 반영합니다.

기술적으로는 다음과 같은 구성으로 구현할 수 있습니다.

| 구성 요소 | 구현 내용 |
| --- | --- |
| 입력 관리 | 요구사항 문서, 변경 요청, 채팅, 회의록 업로드 |
| AI 분석 엔진 | 결정·질문·승인·영향·액션 분류 및 역할별 관점 변환 |
| 프로젝트 메모리 | 회의 결과, 기술 제약, 승인 이력, 반복 쟁점 저장 및 검색 |
| 변경 영향 맵 | 화면, API, 데이터, 테스트, 일정, 담당자 영향 시각화 |
| 합의 잠금 보드 | 확정 항목, 열린 질문, 승인 필요, 착수 조건 관리 |
| 회의 오케스트레이션 | 회의 필요성 판단, 안건 생성, 회의 결과 재구조화 |

실현 가능성도 높습니다. 초기 MVP는 외부 시스템 연동 없이도 요구사항 문서, 채팅 예시, 회의록 텍스트를 기반으로 구현할 수 있습니다. 이후 기업 환경에 맞춰 협업 도구와 연동하면 실제 업무 데이터 기반의 정확도를 높일 수 있습니다. 즉, 처음부터 대규모 엔터프라이즈 시스템을 구축하지 않아도 핵심 가치를 검증할 수 있으며, 단계적으로 확장 가능한 구조입니다.

## 4. 사회적/경제적 파급 효과

RelAi의 기대효과는 문서 작성 시간을 줄이는 것에 그치지 않습니다. 핵심은 요구사항 변경으로 인해 발생하는 재작업, 승인 지연, 회의 반복, 결정 누락을 줄이는 것입니다. AI 코딩 도구가 개발자의 코드 작성 속도를 높이더라도, 요구사항 해석과 협업 조율에서 같은 수준의 시간이 다시 낭비된다면 AI 도입 효과는 제한될 수밖에 없습니다.

첫 번째 효과는 변경 검토 시간 단축입니다. 기존에는 변경 요청이 들어오면 담당자가 관련 문서, 메신저 대화, 회의록, 코드 영향 범위를 직접 확인해야 했습니다. RelAi는 변경 요청을 입력받는 즉시 과거 메모리와 현재 요구사항을 함께 분석하여 영향 범위를 구조화합니다. 이를 통해 초기 검토 시간을 줄이고, 필요한 의사결정을 빠르게 파악할 수 있습니다.

두 번째 효과는 재작업 감소입니다. 변경이 기존 결정과 충돌하는지, 이미 보류하기로 한 기능이 다시 포함되는지, 품질 검수에서 새롭게 영향을 받는 영역이 있는지를 조기에 확인할 수 있습니다. 이는 개발이 상당 부분 진행된 뒤에 다시 범위를 수정하는 상황을 줄이는 데 도움이 됩니다.

세 번째 효과는 승인 지연 감소입니다. 승인권자는 기술적인 세부 내용을 모두 직접 확인하기 어렵습니다. RelAi는 변경 요약, 영향 범위, 보류 시 리스크, 지금 결정해야 할 항목을 브리핑 형태로 제공합니다. 이를 통해 승인권자는 회의록과 채팅을 다시 읽지 않고도 판단에 필요한 핵심 정보를 확인할 수 있습니다.

네 번째 효과는 회의 효율 개선입니다. 모든 논의를 회의로 해결하는 방식은 비효율적이지만, 반대로 회의가 필요한 사안을 채팅으로만 처리하면 결정이 지연됩니다. RelAi는 채팅에서 충돌 신호와 미결정 항목을 감지하여 필요한 경우에만 안건 중심 회의를 제안합니다. 회의 후에는 결정과 액션을 다시 보드와 메모리에 반영하여 회의 결과가 사라지지 않게 합니다.

다섯 번째 효과는 조직 지식 축적입니다. 요구사항 변경 과정에서 반복되는 정책, 기술 제약, 품질 기준, 승인 이력은 조직의 중요한 업무 자산입니다. RelAi는 이를 프로젝트 메모리로 축적하여, 새로운 팀원이 합류하거나 유사한 변경이 다시 발생했을 때 이전 맥락을 재사용할 수 있게 합니다.

사업화 측면에서는 팀 단위 SaaS 구독 모델을 우선 고려할 수 있습니다. 초기 고객은 제품팀, 개발팀, PM 조직, SI/외주 개발 조직, 사내 디지털 전환 조직입니다. 특히 기획, 개발, 품질, 고객사 승인 과정이 반복되는 조직일수록 요구사항 변경 관리 비용이 크기 때문에 도입 명분이 분명합니다.

수익 모델은 다음과 같이 확장할 수 있습니다.

| 수익 모델 | 대상 | 내용 |
| --- | --- | --- |
| 팀 단위 구독 | 제품팀, 개발팀, PM 조직 | 월 구독 형태로 프로젝트별 협업 공간 제공 |
| 프로젝트 단위 과금 | 외주/고객사 협업 프로젝트 | 프로젝트 기간 동안 변경 관리와 의사결정 기록 제공 |
| 협업 패키지 | 기획사, 개발사, QA 조직 | 요구사항 변경, 품질 피드백, 승인 브리핑 기능 제공 |
| 엔터프라이즈 플랜 | 중견·대기업 | 보안, 권한 관리, 내부 시스템 연동, 감사 로그 제공 |

RelAi는 AI로 더 많은 문서를 만드는 서비스가 아닙니다. AI 시대에 더 빠르게 발생하는 요구사항 변경과 해석 차이를 팀 단위의 합의 구조로 바꾸는 업무혁신 시스템입니다. 개인의 AI 활용을 넘어, 기획-개발-품질-승인자가 함께 일하는 방식을 AX 관점에서 재설계하는 것이 RelAi의 목표입니다.
