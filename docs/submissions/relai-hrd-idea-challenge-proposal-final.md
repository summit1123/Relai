# Relai

## 요구사항 변경과 협업 맥락을 정렬하는 AI 업무혁신 시스템

### 1. 제안 배경 및 필요성

기업 현장에서는 기획, 개발, 품질 검수, 승인 과정이 순서대로만 진행되지 않습니다. 기획 초안이 나오면 개발이 시작되고, 개발 중간에 정책이나 우선순위가 바뀌며, 품질 검수 과정에서 다시 예외 케이스와 수정 사항이 발견됩니다. 이 과정은 특정 직군의 역량 부족으로만 보기 어렵습니다. 실제 업무에서는 요구사항이 계속 구체화되고, 이해관계자가 늘어나며, 출시 일정과 품질 기준이 함께 움직이기 때문입니다.

최근에는 생성형 AI 도구의 확산으로 개인의 업무 속도는 더 빨라지고 있습니다. 기획자는 AI로 사용자 시나리오와 정책 문구를 빠르게 만들고, 개발자는 AI로 코드와 구현 방식을 빠르게 설계하며, 품질 담당자는 AI로 테스트 케이스와 예외 상황을 확장합니다. 그러나 각자가 AI를 사용하는 목적과 방식이 다르기 때문에, 같은 요구사항도 기획 의도, 구현 범위, 품질 기준이 서로 다른 방향으로 빠르게 확장될 수 있습니다.

문제는 AI를 쓰지 않는 것이 아니라, 각자가 AI로 만든 판단과 산출물이 팀 단위의 합의 구조로 다시 정렬되지 않는다는 점입니다. Microsoft Work Trend Index 2024에 따르면 지식근로자의 75%가 업무에서 AI를 사용하고, AI 사용자 중 78%는 개인적으로 선택한 AI 도구를 업무에 활용하고 있습니다. Gallup 조사에서도 미국 직장인의 업무상 AI 사용은 2023년 21%에서 2025년 40%로 증가했습니다. 반면 조직이 AI를 통합하고 있다는 응답은 44%였지만, 명확한 AI 전략을 전달받았다는 응답은 22%에 그쳤습니다.

개발 현장에서도 비슷한 간극이 나타납니다. Atlassian의 2025 Developer Experience 조사에서는 개발자 68%가 AI로 주 10시간 이상을 절약한다고 답했지만, 동시에 개발자 50%는 정보 탐색, 도구 전환, 협업 문제 등 조직 비효율로 주 10시간 이상을 잃고 있었습니다. 개발자가 실제 코딩에 쓰는 시간은 약 16%에 불과하며, 나머지 시간은 요구사항 확인, 회의, 승인, 문서 확인, 협업 조율 등 코드 주변 업무에 쓰입니다.

의사소통 문제 역시 재작업으로 이어집니다. Atlassian/Loom의 2026년 조사에서는 근로자 80%가 글로 된 메시지를 오해해 불필요하거나 다시 해야 하는 일을 경험했다고 응답했습니다. PMI 2021 조사에서도 프로젝트 실패 원인으로 요구사항 수집 부실 40%, 프로젝트 목표 변경 38%, 커뮤니케이션 부실 28%가 제시되었습니다.

따라서 AI 시대의 업무혁신은 단순히 개인에게 더 많은 AI 도구를 제공하는 데서 끝나서는 안 됩니다. 기획, 개발, 품질, 승인자가 각자 AI로 만든 판단을 하나의 프로젝트 맥락 안에서 정렬하고, 변경 사항이 어디까지 영향을 미치는지 빠르게 파악할 수 있는 협업 운영 구조가 필요합니다.

### 2. 아이디어 개요

Relai는 요구사항 변경, 팀 채팅, 회의 기록, AI 산출물을 하나의 프로젝트 메모리로 통합하고, 이를 결정·질문·승인·영향·액션 단위로 구조화하는 AI 업무혁신 시스템입니다.

기존 협업 도구는 문서, 티켓, 채팅, 회의록을 각각 저장하는 데 강점이 있습니다. 그러나 변경이 발생했을 때 어떤 결정이 다시 열렸는지, 어떤 화면과 API가 영향을 받는지, 누가 승인해야 하는지, 이전 회의에서 어떤 조건이 합의되었는지를 한 번에 보여주지는 못합니다. Relai는 이 흩어진 맥락을 하나의 흐름으로 연결합니다.

Relai의 주요 사용자는 PM/PO, 서비스 기획자, 개발 리드, 품질 담당자, 승인권자입니다. 이들은 같은 프로젝트 안에 있지만 서로 다른 관점으로 요구사항을 바라봅니다. Relai는 하나의 변경 요청을 각 역할의 실행 언어로 다시 정리하고, 변경 영향과 합의 상태를 함께 보여줍니다.

Relai의 기본 흐름은 다음과 같습니다.

| 단계 | 내용 |
| --- | --- |
| 요구사항 입력 | 기획 초안, 변경 요청, 정책 수정, 품질 피드백을 등록합니다. |
| 관점 정렬 | AI가 같은 내용을 기획, 개발, 품질 관점으로 재구성합니다. |
| 합의 상태 분리 | 잠긴 결정, 열린 질문, 승인 필요 항목, 착수 가능 조건을 분류합니다. |
| 영향 분석 | 변경이 화면, API, 데이터, 테스트, 일정, 담당자에게 미치는 영향을 정리합니다. |
| 프로젝트 메모리화 | 이전 채팅, 회의, 결정, 기술 제약을 다음 판단에 활용할 수 있게 저장합니다. |
| 회의 제안 | 채팅만으로 합의가 어려운 경우 AI가 안건과 참석자를 제안합니다. |

Relai는 단순히 회의록을 요약하거나 기획서를 작성하는 도구가 아닙니다. 핵심은 AI가 만든 개별 산출물을 팀의 실행 가능한 합의 구조로 다시 정렬하는 것입니다.

### 3. 서비스 이용 시나리오

예를 들어, 기획자가 “관리자 승인 후 결제를 확정하는 정책을 이번 릴리즈에 포함하고 싶다”고 요청했다고 가정합니다. 일반적인 협업 방식에서는 이 변경이 단순한 정책 수정처럼 보일 수 있습니다. 그러나 실제 개발 현장에서는 결제 완료 화면, 결제 확정 API, 데이터베이스 필드, 승인 전후 테스트 케이스, QA 일정, 담당자 배정, 승인권자 판단까지 함께 영향을 받을 수 있습니다.

Relai는 이 변경 요청을 단순 텍스트로 두지 않고 변경 영향 맵으로 전환합니다. AI는 기존 요구사항, 이전 회의 기록, 팀 채팅, 프로젝트 메모리를 함께 참고하여 무엇이 이미 확정된 결정인지, 무엇이 다시 열린 항목인지, 누가 결정해야 하는지를 구조화합니다.

사용자는 변경 요청을 등록한 뒤 다음 정보를 한 화면에서 확인할 수 있습니다.

| 구분 | 예시 |
| --- | --- |
| 변경 요청 | 관리자 승인 후 결제 확정으로 정책 변경 |
| 영향 화면 | 결제 완료 페이지, 관리자 승인 화면 |
| 영향 API | payment-confirm endpoint |
| 영향 데이터 | approval_status 필드 추가 |
| 테스트 영향 | 승인 전/후 결제 시나리오 추가 |
| 일정 영향 | QA 기간 2일 증가 가능 |
| 담당자 | 백엔드 개발자, QA 담당자 |
| 승인 필요 | PO 또는 프로젝트 승인권자 확인 필요 |

이후 Relai는 합의 잠금 보드에서 이미 확정된 결정과 다시 논의해야 할 질문을 분리합니다. 채팅만으로 합의가 어렵다고 판단되면 AI가 회의 필요성을 제안하고, 회의 제목, 참석자, 핵심 안건, 사전 확인 자료를 함께 생성합니다.

### 4. 구현 방식 및 기술 구조

Relai의 핵심 구조는 프로젝트 메모리입니다. 요구사항 문서, 변경 요청, 팀 채팅, 회의 기록, 녹음 파일, AI 산출물은 모두 프로젝트 컨텍스트로 유입됩니다. AI는 이를 단순 저장하지 않고 결정, 질문, 승인, 영향, 액션, 근거, 메모리로 구조화합니다.

이 구조가 필요한 이유는 요구사항 변경이 현재 문장 하나만으로 판단되지 않기 때문입니다. 과거 회의에서 “이 기능은 2차 릴리즈로 미룬다”고 합의했을 수 있고, 개발 리드가 “결제 API 변경 시 QA 기간이 2일 더 필요하다”고 말했을 수 있으며, 승인권자가 “운영 리스크 때문에 수동 승인 옵션은 유지해야 한다”고 남겼을 수 있습니다. 이런 맥락이 사라지면 팀은 같은 논의를 반복하게 됩니다.

Relai는 새 변경 요청이 들어왔을 때 이전 대화와 회의 기록을 함께 참고합니다. 그리고 채팅에서 기획자와 개발자의 판단이 충돌하거나, 품질 기준과 출시 일정 사이의 선택이 필요한 경우 AI가 회의 필요성을 판단합니다. 이때 회의는 단순 일정 예약이 아니라 결정해야 할 안건 중심으로 구성됩니다.

회의가 끝난 뒤에는 녹음 또는 회의록이 다시 AI에 의해 구조화됩니다. 새 결정, 열린 질문, 승인 필요 항목, 액션 아이템, 메모리 저장 항목이 생성되고, 이는 다음 변경 검토의 근거가 됩니다. 이 과정을 통해 Relai는 일회성 분석 도구가 아니라 팀의 맥락을 축적하는 협업 운영 시스템이 됩니다.

| 처리 영역 | 주요 기능 |
| --- | --- |
| 입력 수집 | 요구사항 문서, 채팅, 회의록, 녹음, AI 산출물 수집 |
| 구조화 | 결정, 질문, 승인, 영향, 액션으로 분류 |
| 맥락 저장 | 반복 쟁점, 기술 제약, 승인 이력, 품질 기준을 메모리화 |
| 영향 분석 | 변경 요청이 업무 영역별로 미치는 영향 추정 |
| 회의 오케스트레이션 | 충돌 감지 시 회의 제목, 참석자, 안건 제안 |
| 후속 반영 | 회의 결과를 보드와 프로젝트 메모리에 재반영 |

### 5. 기대효과 및 사업화 방안

Relai의 기대효과는 문서 작성 시간을 줄이는 데 그치지 않습니다. 핵심은 요구사항 변경으로 인해 발생하는 재작업, 승인 지연, 회의 반복, 결정 누락을 줄이는 것입니다.

개발자가 AI로 코드 작성 시간을 절약하더라도, 요구사항 확인과 협업 조율에서 같은 수준의 시간을 다시 잃는다면 AI 도입 효과는 제한될 수밖에 없습니다. Relai는 개발 자체보다 개발 주변에서 발생하는 요구사항 정렬, 정보 탐색, 회의, 승인, 품질 피드백의 병목을 줄이는 데 초점을 둡니다.

| 기대효과 | 내용 |
| --- | --- |
| 변경 검토 시간 단축 | 변경 요청이 들어왔을 때 영향 범위를 빠르게 구조화합니다. |
| 재작업 감소 | 기존 결정과 충돌하는 변경을 조기에 발견합니다. |
| 승인 지연 감소 | 승인권자가 판단해야 할 항목과 리스크를 명확히 확인합니다. |
| 회의 효율 개선 | 필요한 회의만 안건 중심으로 제안합니다. |
| 결정 누락 감소 | 회의 후 결정과 액션을 다시 보드와 메모리에 반영합니다. |
| 조직 지식 축적 | 반복되는 정책, 기술 제약, 품질 기준을 프로젝트 메모리로 남깁니다. |

사업화는 팀 단위 SaaS 구독 모델을 기본으로 합니다. 초기 고객은 기획-개발-품질 협업이 잦은 제품팀, SI/외주 개발 조직, 사내 디지털 전환 조직입니다. 프로젝트 단위 과금 모델도 적용할 수 있으며, 외주 프로젝트에서는 변경 요청과 승인 이력을 명확히 남기는 계약 관리 도구로 확장할 수 있습니다.

| 수익 모델 | 대상 | 내용 |
| --- | --- | --- |
| 팀 단위 구독 | 제품팀, 개발팀, PM 조직 | 월 구독 형태로 프로젝트별 협업 공간 제공 |
| 프로젝트 단위 과금 | 외주/고객사 협업 프로젝트 | 프로젝트 기간 동안 변경 관리와 의사결정 기록 제공 |
| 협업 패키지 | 기획사, 개발사, QA 조직 | 요구사항 변경, 품질 피드백, 승인 브리핑 기능 제공 |

향후에는 Jira, Notion, Slack, Teams와 연동하여 실제 업무 도구 안의 문서와 대화를 연결할 수 있습니다. 또한 코드 저장소 기반 변경 영향 분석, QA 테스트 케이스 자동 연결, 회의 녹음 자동 구조화 기능으로 확장할 수 있습니다.
