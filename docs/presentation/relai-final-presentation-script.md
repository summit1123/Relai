## Slide 2. 문제 정의

### 슬라이드 제목
문제는 변경 그 자체가 아니라, 합의 구조의 부재입니다

### 화면에 넣을 핵심 문장
- 기획이 바뀌면 개발이 흔들립니다
- 품질 검수가 들어오면 다시 수정이 열립니다
- 문제는 역량 부족보다 정렬 방식의 부재입니다

### 시각 구성
- 좌에서 우로 `기획자 -> 개발자 -> QA -> 승인권자`
- 각 단계 사이에 `변경`, `재작업`, `승인 지연`, `회의 반복` 태그

### 발표 멘트
현업에서는 기획 초안이 나오고 바로 개발이 시작되지만, 개발 중간에 다시 기획이 수정되고, 품질 검수 과정에서 예외 케이스가 발견되며 수정이 다시 열립니다.  
이 과정은 어느 한 직군이 부족해서 생기는 문제가 아닙니다. 기획자는 사용자 가치 중심으로, 개발자는 구현 가능성과 일정 중심으로, QA는 안정성과 예외 상황 중심으로 보기 때문에 같은 요구사항도 서로 다르게 이해될 수 있습니다.  
결국 문제는 변경 자체보다, 이 변경을 팀이 같은 기준으로 다시 정렬하는 구조가 없다는 데 있습니다.

## Slide 3. 왜 지금인가

### 슬라이드 제목
AI는 빨라졌지만, 팀의 정렬 방식은 그대로입니다

### 화면에 넣을 수치
- 지식근로자 75% AI 사용
- 개발자 68% AI로 시간 절약
- 근로자 80% 메시지 오해로 재작업 경험
- 요구사항 수집 부실 40%

### 시각 구성
- 좌측: `개인 생산성 상승`
- 우측: `팀 정렬 병목 유지`
- 가운데 대비 화살표

### 발표 멘트
지금은 AI를 쓰느냐 마느냐의 문제가 아닙니다. 이미 많은 사람들이 AI를 사용하고 있습니다.  
문제는 각자가 AI로 더 빨리 문서를 만들고, 더 빨리 코드를 짜고, 더 빨리 테스트를 확장하는 동안, 팀은 여전히 문서와 메신저, 회의에 의존해 합의를 맞추고 있다는 점입니다.  
즉 개인 단위의 생산성은 올라갔지만, 조직 단위의 정렬 방식은 그대로 남아 있습니다. 이 간극이 앞으로 더 큰 재작업 비용으로 돌아올 수 있다고 봤습니다.

## Slide 4. 인사이트

### 슬라이드 제목
진짜 AX는 개인이 아니라 조직의 의사소통 속도를 높이는 것입니다

### 화면에 넣을 핵심 문장
개인의 AI 활용 -> 조직의 합의 구조로 연결되어야 한다

### 발표 멘트
저희는 여기서 하나의 인사이트를 얻었습니다.  
진정한 AX는 개인이 AI를 잘 쓰는 것에서 끝나지 않습니다. 기획자, 개발자, QA, 승인권자가 각자 AI로 만든 판단을 하나의 프로젝트 맥락 안에서 다시 정렬하고, 변경의 영향이 어디까지 퍼지는지 빠르게 확인할 수 있어야 합니다.  
RelAi는 바로 이 조직 단위의 정렬 구조를 만드는 시스템입니다.

## Slide 5. RelAi가 하는 일

### 슬라이드 제목
RelAi는 변경을 실행 가능한 합의 구조로 바꿉니다

### 화면에 넣을 구조
- 입력: 요구사항 문서 / 회의 기록 / 채팅 / 변경 요청
- 출력: 결정 / 질문 / 승인 / 영향 / 액션 / 메모리

### 발표 멘트
RelAi는 여러 곳에 흩어져 있는 협업 맥락을 하나로 모읍니다.  
입력은 요구사항 문서, 팀 채팅, 회의 기록, 변경 요청입니다.  
그리고 AI는 이를 단순히 요약하는 것이 아니라, 결정, 질문, 승인, 영향, 액션, 메모리라는 실행 단위로 구조화합니다.  
즉 RelAi는 문서를 더 많이 만드는 도구가 아니라, 이미 존재하는 논의를 팀이 실제로 움직일 수 있는 구조로 바꾸는 도구입니다.

## Slide 6. 핵심 플로우

### 슬라이드 제목
초안부터 변경, 영향, 회의까지 하나의 흐름으로 이어집니다

### 화면에 넣을 플로우
초안 등록 -> 역할별 해석 변환 -> 합의 잠금 보드 -> 변경 영향 맵 -> 프로젝트 메모리 -> AI 회의 개설

### 추천 자산
- [relai-ppt-storyboard-concept.png](/Users/gimdonghyeon/Desktop/Relai/docs/presentation/assets/relai-ppt-storyboard-concept.png)

### 발표 멘트
RelAi의 흐름은 단순합니다.  
먼저 요구사항 초안이나 변경 요청을 입력합니다.  
그러면 AI가 같은 내용을 기획, 개발, 품질 관점으로 다시 정리합니다.  
이후 무엇이 확정인지, 무엇이 아직 열린 질문인지, 어떤 항목이 승인 필요한지 합의 잠금 보드에서 나뉩니다.  
변경 요청이 들어오면 영향 맵으로 화면, API, 데이터, 테스트, 일정, 담당자까지 어디에 영향이 퍼지는지 확인합니다.  
그리고 모든 과정은 프로젝트 메모리로 누적되어 다음 판단의 근거가 됩니다. 필요하면 AI가 회의 개설까지 제안합니다.

## Slide 8. 차별성

### 슬라이드 제목
기존 도구는 기록을 남기고, RelAi는 다음 실행을 정렬합니다

### 비교 표
| 구분 | 기존 도구 | RelAi |
| --- | --- | --- |
| Mantis/Jira | 이슈를 남김 | 변경이 누구에게 어떤 의미인지 해석 |
| Confluence/Notion | 문서를 남김 | 문서와 대화를 프로젝트 메모리로 연결 |
| 메신저/회의 | 대화를 남김 | 결정, 질문, 승인, 액션으로 구조화 |

### 발표 멘트
기존 도구들은 분명 필요합니다. Mantis나 Jira는 이슈를 남기고, Confluence나 Notion은 문서를 남기고, 메신저는 빠른 대화를 가능하게 합니다.  
하지만 변경이 생겼을 때 그 기록을 다시 해석해서, 이번 변경이 누구에게 어떤 의미인지, 어떤 결정이 다시 열렸는지, 무엇이 승인 필요 항목인지까지 정렬해주지는 못합니다.  
RelAi의 차별성은 바로 여기에 있습니다. 기록을 남기는 것이 아니라, 그 기록을 바탕으로 다음 실행을 정렬합니다.

## Slide 9. 실현 가능성과 검증

### 슬라이드 제목
RelAi는 이미 MVP 단계에서 핵심 흐름을 검증 중입니다

### 화면에 넣을 포인트
- 웹 기반 MVP 개발 중
- 핵심 기능 완성 단계
- 실제 업무 흐름 기준 검증
- 향후 외부 도구 연동 확장 가능

### 발표 멘트
RelAi는 아이디어 수준에 머물러 있지 않습니다. 현재 웹 기반 MVP를 개발 중이며, 핵심 기능은 이미 완성 단계에 가깝습니다.  
요구사항 초안 등록, 역할별 정리, 변경 영향 맵, 프로젝트 메모리, AI 회의 개설 흐름을 실제 업무 시나리오 기준으로 검증하고 있습니다.  
초기에는 독립형 시스템으로 시작하지만, 향후에는 Jira, Notion, Slack, Teams, GitHub, Mantis, Confluence와의 연동을 통해 실제 조직 안으로 더 자연스럽게 들어갈 수 있습니다.

## Slide 10. 사업성 및 기대효과

### 슬라이드 제목
재작업을 줄이고, 승인 속도를 높이고, 조직의 AX 기반이 됩니다

### 화면에 넣을 기대효과
- 재작업 감소
- 변경 검토 시간 단축
- 승인 지연 감소
- 회의 후 결정 누락 감소

### 화면에 넣을 수익모델
- 팀 단위 구독
- 프로젝트 단위 과금
- 엔터프라이즈 플랜
- 부서 확장 패키지

### 발표 멘트
RelAi의 효과는 문서 정리에 있지 않습니다.  
변경 검토 시간을 줄이고, 기존 결정과 충돌하는 요청을 빨리 발견하고, 승인권자가 필요한 판단을 더 빠르게 내릴 수 있게 하는 데 있습니다.  
사업화는 팀 단위 SaaS 구독으로 시작할 수 있습니다. 초기 고객은 제품팀, 개발팀, PM 조직, SI/외주 협업팀입니다. 이후에는 엔터프라이즈 플랜과 부서 확장 패키지로 확장할 수 있습니다.  
결국 RelAi는 개발 직군만의 도구가 아니라, 조직의 의사소통 속도를 AI 흐름에 맞게 끌어올리는 협업 운영 플랫폼으로 발전할 수 있습니다.
