## Slide 2. 문제 정의

### 제목
문제는 변경 그 자체가 아니라, 합의 구조의 부재입니다

### 화면 핵심 문장
- 기획이 바뀌면 개발이 흔들립니다
- 품질 검수가 들어오면 다시 수정이 열립니다
- 문제는 역량 부족보다 정렬 구조 부재입니다

### 시각 구성
- `기획자 -> 개발자 -> QA -> 승인권자`
- 각 단계 사이에 `변경`, `재작업`, `승인 지연`, `회의 반복`

### 시간
1분

### 발표 멘트
현업에서는 기획 초안이 나오면 개발이 바로 시작되지만, 개발 중간에 정책이나 우선순위가 바뀌고, 품질 검수 과정에서 다시 예외 케이스와 수정 요청이 열립니다.  
이건 어느 한 직군의 역량 부족으로 보기 어렵습니다. 기획자는 사용자 가치 중심으로, 개발자는 구현 가능성과 일정 중심으로, QA는 안정성과 예외 상황 중심으로 보기 때문에 같은 요구사항도 서로 다른 언어로 해석되기 때문입니다.  
결국 문제는 변경 자체보다, 이 변경을 팀이 같은 기준으로 다시 정렬할 구조가 없다는 데 있습니다.

## Slide 3. 왜 지금인가

### 제목
AI는 빨라졌지만, 팀의 정렬 방식은 그대로입니다

### 화면 수치
- 지식근로자 75% AI 사용
- 개발자 68% AI로 시간 절약
- 근로자 80% 메시지 오해로 재작업 경험
- 요구사항 수집 부실 40%

### 시각 구성
- 좌측: `개인 생산성 상승`
- 우측: `팀 정렬 병목 유지`

### 시간
1분 10초

### 발표 멘트
이제는 AI를 쓰느냐 안 쓰느냐의 문제가 아닙니다. 이미 많은 사람들이 AI를 쓰고 있습니다.  
문제는 각자가 AI로 문서를 더 빨리 만들고, 코드를 더 빨리 작성하고, 테스트를 더 빨리 확장하는 동안, 팀은 여전히 문서와 메신저, 회의에 의존해 합의를 맞추고 있다는 점입니다.  
즉 개인 단위의 속도는 올라갔지만, 조직 단위의 정렬 방식은 그대로 남아 있습니다. 저희는 이 간극이 앞으로 더 큰 재작업 비용을 만들 것이라고 판단했습니다.

### 화면 핵심 문장
개인의 AI 활용 -> 조직의 합의 구조로 연결되어야 한다

### 시간
45초

저희는 여기서 하나의 인사이트를 얻었습니다.  
진정한 AX는 개인이 AI를 잘 쓰는 것에서 끝나지 않습니다. 기획자, 개발자, QA, 승인권자가 각자 AI로 만든 판단을 하나의 프로젝트 맥락 안에서 다시 정렬하고, 변경의 영향이 어디까지 퍼지는지 빠르게 확인할 수 있어야 합니다.  
RelAi는 바로 이 조직 단위의 정렬 구조를 만드는 시스템입니다.

## Slide 5. RelAi 정의

### 제목
RelAi는 변경을 실행 가능한 합의 구조로 바꿉니다

### 화면 구조
- 입력: 요구사항 문서 / 회의 기록 / 채팅 / 변경 요청
- 출력: 결정 / 질문 / 승인 / 영향 / 액션 / 메모리

### 시간
55초

### 발표 멘트
RelAi는 여러 곳에 흩어져 있는 협업 맥락을 하나로 모읍니다.  
입력은 요구사항 문서, 팀 채팅, 회의 기록, 변경 요청입니다.  
그리고 AI는 이를 단순히 요약하는 것이 아니라, 결정, 질문, 승인, 영향, 액션, 메모리라는 실행 단위로 구조화합니다.  
즉 RelAi는 문서를 더 많이 만드는 도구가 아니라, 이미 존재하는 논의를 팀이 실제로 움직일 수 있는 구조로 바꾸는 도구입니다.

### 제목

### 화면 플로우
초안 등록 -> 역할별 해석 변환 -> 합의 잠금 보드 -> 변경 영향 맵 -> 프로젝트 메모리 -> AI 회의 개설

### 발표 멘트
RelAi의 흐름은 여섯 단계로 이해하시면 됩니다.

변경 요청이 들어오면 영향 맵으로 화면, API, 데이터, 테스트, 일정, 담당자까지 어디에 영향이 퍼지는지 확인합니다.  
그리고 모든 과정은 프로젝트 메모리로 누적되어 다음 판단의 근거가 됩니다. 필요하면 AI가 회의 개설까지 제안합니다.

## Slide 7. 실제 사용 시나리오

### 제목
변경 요청 하나가 어떻게 정렬되는가

### 화면 예시
- 변경 요청: `관리자 승인 후 결제 확정으로 정책 변경`
- 영향: 화면 / API / 데이터 / 테스트 / 일정 / 담당자 / 승인권자

### 시간
1분

### 발표 멘트
예를 들어, 기획자가 관리자 승인 후 결제를 확정하는 정책을 이번 릴리즈에 넣고 싶다고 요청했다고 가정해보겠습니다.  
일반적으로는 단순 정책 수정처럼 보일 수 있지만, 실제로는 결제 완료 화면, 결제 확정 API, 데이터베이스 필드, QA 시나리오, 일정, 승인권자 판단까지 함께 영향을 받습니다.  
RelAi는 이 요청을 단순 텍스트로 두지 않고, 누가 무엇을 다시 봐야 하는지와 어떤 결정이 다시 열렸는지까지 구조화합니다.

## Slide 9. 차별성

### 제목
기존 도구는 기록을 남기고, RelAi는 다음 실행을 정렬합니다

### 비교 표
| 구분 | 기존 도구 | RelAi |
| --- | --- | --- |
| Mantis/Jira | 이슈를 남김 | 변경이 누구에게 어떤 의미인지 해석 |
| Confluence/Notion | 문서를 남김 | 문서와 대화를 프로젝트 메모리로 연결 |
| 메신저/회의 | 대화를 남김 | 결정, 질문, 승인, 액션으로 구조화 |

### 시간
1분

### 발표 멘트
기존 도구들은 분명 필요합니다. 이슈를 남기고, 문서를 축적하고, 빠른 대화를 가능하게 해줍니다.  
하지만 변경이 생겼을 때 그 기록을 다시 해석해서, 이번 변경이 누구에게 어떤 의미인지, 어떤 결정이 다시 열렸는지, 무엇이 승인 필요 항목인지까지 정렬해주지는 못합니다.  
RelAi의 차별성은 바로 이 지점입니다. 기록을 남기는 것이 아니라, 그 기록을 바탕으로 다음 실행을 정렬합니다.

## Slide 10. 실현 가능성 및 사업성

### 시간
1분

### 발표 멘트
RelAi는 아이디어 수준에 머물러 있지 않습니다. 웹 기반 MVP를 개발 중이며, 핵심 기능은 이미 완성 단계에 가깝습니다.  
요구사항 초안 등록, 역할별 정리, 변경 영향 맵, 프로젝트 메모리, AI 회의 개설 흐름을 실제 업무 시나리오 기준으로 검증하고 있습니다.  
사업화는 팀 단위 SaaS 구독으로 시작할 수 있습니다. 초기 고객은 제품팀, 개발팀, PM 조직, 그리고 SI나 외주 협업팀입니다. 이후에는 엔터프라이즈 플랜과 부서 확장 패키지로 확장할 수 있습니다.

## Appendix C. 프로젝트 메모리 설명

## Appendix D. 기존 도구와의 관계

### 제목

## Appendix G. 예상 질문과 짧은 답변

### Q1. 왜 기존 협업툴로 안 되나?
기존 도구는 기록을 남기는 데 강하지만, 변경이 생겼을 때 그 기록을 다시 해석해 다음 실행을 정렬해주지는 못합니다. RelAi는 바로 그 정렬 레이어를 제공합니다.

### Q2. AI가 오판하면 더 위험하지 않나?
RelAi는 자동 실행 도구가 아니라 판단 지원 도구입니다. 잠긴 결정, 열린 질문, 승인 필요 항목을 분리하고 사람 승인 구조를 유지합니다.

### Q3. 가장 먼저 돈 내고 쓸 팀은 어디인가?
요구사항 변경과 직군 간 협업이 잦은 제품팀, 개발팀, PM 조직, 그리고 SI/외주 프로젝트가 가장 먼저 쓸 가능성이 높습니다.

### Q4. MVP가 어디까지 되어 있나?
핵심 흐름인 초안 등록, 역할별 정리, 변경 영향 맵, 프로젝트 메모리, AI 회의 개설 구조는 이미 MVP 수준에서 구현 및 검증 중입니다.

### Q5. 장기적으로는 어떤 서비스가 되나?
RelAi는 단순한 기능형 AI 툴이 아니라, 조직의 요구사항 변경과 의사결정 메모리를 관리하는 AX 협업 플랫폼으로 확장할 수 있습니다.
